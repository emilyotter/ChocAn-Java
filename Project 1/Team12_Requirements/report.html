Individual Project Requirements

In project one we were tasked with creating a use case diagram, descriptions for each use case, and a glossary of some key terms. For the use case diagram, our team each created their own diagram, then we all came together and used the best elements for each of ours to construct the one you will find in this report. For the use case description part of the project, the use cases were written in a Google document, reviewed, and then added to the report file. Each description was designed to accurately describe what the use case does, and how it would function. The glossary section of the project consists of key terms that are used repeatedly throughout the project and this report. As mentioned before, each team member has their own use case diagram. Each of these is embedded in its own FirstLastTeam12Report.html file. All of the images are named in the format of FirstLastUML.png and are included in the UMLDiagrams_PNG directory. The team's use case diagram is embedded in this file: report.html. This diagram can also be found under the UMLDiagrams_PNG directory, and it is named Team12UML.png. The submitter for the reports.html file is Maddox Guthrie.

Use Case Diagram

Use Case Descriptions

Use Case: Authenticate member
Context: Upon entering a member card into the system, the terminal verifies the member to receive health care service.
Actors: ChocAn Data Center, Provider Terminal
Main Success Scenario:
  1. Member card is provided into terminal card reader
  2. Terminal dials ChocAn Data Center
  3. ChocAn Data Center computer verifies the member number
    1. If the number is valid, the word Validated appears on the one-line display.
Extensions:
  1. If the number is invalid, the reason is displayed, such as Invalid number or Member suspended; the latter message indicates that fees are owed (that is, the member has not paid membership fees for at least a month) and member status has been set to suspended.
Use case: Provider Directory
Context: A use case of the provider terminal, that the provider can use to perform specific functions such as looking up the service code for a service.
Actors: Provider Terminal
Main Success Scenario:
  1. Provider enters their provider number to authenticate. The provider accesses other commands from the terminal, represented as other use cases.
Use case: Authenticate provider
Context: Authenticate the provider using the provider number
Actors: Provider Terminal
Main Success Scenario:
  1. Provider enters their provider number to authenticate.
Use case: List directory
Context: At any time a provider can list the provider directory, which will create a list of all the services, the corresponding codes, and the fees for each service.
Actors: Provider Terminal
Main Success Scenario:
  1. Provider requests the list of the directory
  2. An alphabetically ordered list of service names and corresponding service codes and fees is generated
  3. The list is emailed to the provider as an attachment
Use case: Check service code
Context: Checks whether the inputted service code is valid.
Actors: Provider Terminal
Main Success Scenario:
  1. The provider enters a service code.
  2. The corresponding service to the code is displayed.
    1. Provider is asked to verify if this is the correct service
Extensions:
  1. An error is displayed if an incorrect error code was inputted.
Use Case: Manage Billing
Context: The provider intends to bill ChocAN for services provided.
Actors: Provider Terminal
Main Success Scenario:
  1. Provider slides card though card reader on terminal.
  2. The system validates the member.
    1. Terminal dials ChocAn Data Center.
    2. ChocAn Data Center verifies member number.
  3. Provider keys in the date of service in the format MM-DD-YYYY.
  4. Provider looks up six-digit service code in Provider Directory.
  5. Provider keys in service code.
    1. Software product displays name of service corresponding to code.
    2. Provider verifies correct service.
  6. Software product looks up fee to be paid for service.
    1. Software displays fee on provider terminal.
  7. Provider enters information.
    1. Provider enters current date and time on form, member name and number, service code, and fee to be paid.
  8. At the end of the week, provider totals fees to verify amount paid by ChocAn.
Use Case: Accounting Procedure
Context: The main accounting procedure run at ChocAn Data Center at midnight on Fridays.
Actors: ChocAn Data Center
Main Success Scenario:
  1. Accounting procedure is run at ChocAn Data Center at midnight on Friday.
  2. The week’s file of services is read.
  3. Reports are generated.
  4. Reports are sent out.
  5. Reports are sent out.
  6. A record of electronic funds transfer (EFT) data is written to a disk.
    1. Banking computers ensure provider's account is credited.
  7. A summary report is given to the manager for accounts payable.
    1. The report lists every provider to be paid, the number of consultations, and total fees.
  8. A totals sheet of number of providers who gave services, number of consultations, and overall fees is printed.
Extensions:
  1. Report is requested individually by the request of ChocAn manager at any time during the week.
Use Case: Send Reports
Context: A part of the main accounting procedure where reports are sent to members, providers, and managers.
Actors: ChocAn Data Center
Main Success Scenario:
  1. Reports are generated.
  2. Reports are sent to members.
    1. Reports are both printed and emailed as an attachment.
  3. Reports are sent to providers.
    1. Reports are both printed and emailed as an attachment.
Use Case: Generate Reports
Context: A part of the main accounting procedure where reports are created.
Actors: ChocAn Data Center
Main Success Scenario:
  1. Files are read containing information on the week's services.
  2. Reports sent to members are sorted in order of service date.
    1. Reports includes member name, number, street address, city, state, ZIP code and service data, provider name, and service name for each service provided.
  3. Reports are sent to providers are sorted in order of when the data was recieved by the computer.
    1. Report contains same information entered on provider's form.
  4. Summary including number of consultations with members and the total fees for the week are written at the bottom of the report.
Use Case: Interactive Operations
Context: The ChocAn Data Center software can be run in interactive mode, to give the operator options to update member and provider records.
Actors: Operators, ChocAn Data Center, Acme Accounting Services
Main Success Scenario:
  1. The ChocAn Data Center must be in interactive mode.
  2. The operator may update provider details
    1. Change records
    2. Add provider
    3. Delete provider
  3. The operator may update member details
    1. Change records.
    2. Add member.
    3. Delete Member.
    4. The Acme Accounting Services may suspend members due to unsatisfied payments.
Use Case: Add member
Context: Operator Intends to add a new member to the database.
Actors: Operator, Acme Accounting Service
Main Success Scenario:
  1. The operator selects “add member”
  2. Operator is prompted to fill up the member details
  3. The system communicates with the Acme Accounting Service to record the member details.
    1. The system receives the confirmation from Acme Accounting Service.
  4. After completion, the operator is informed about the status of the member and taken to the Interactive terminal.
Extensions:
    1. If the customer already exists, the membership feed is checked, and the operator is informed about the member status.
    2. If the customer does not exist, a new member is added to the database, and the membership fees are collected.
Use Case: Delete member
Context: Operator Intends to delete an existing member from the database.
Actors: Operator, Acme Accounting Service
Main Success Scenario:
  1. The operator selects “delete member”
  2. The operator is prompted for member details like member ID
  3. The system communicates with the Acme Accounting Service to change the member status.
    1. The system receives the confirmation from Acme Accounting Service.
  4. The system deletes the member records from the database.
  5. After completion, the operator is informed about the status of the member and taken to the Interactive terminal.
Extensions:
    1. If the member does not exist, an error is raised, and the operator is informed.
    2. If the member exists, Acme Accounting Service is informed to disable membership fees collection, and delete member personal info.
Use Case: Edit member records
Context: Operator or Acme Accounting Service intends to edit a member's record from the database.
Actors: Operator, Acme Accounting Service
Main Success Scenario:
  1. Operator intends to change the member records.
  2. The operator is displayed current member info and prompted for changes to the records.
  3. The changes are verified and reconfirmed by the operator.
  4. The changes are saved to the database.
  5. The operator is informed of the changes and taken to interactive operations for further commands.
Additional Success Scenario:
  1. Acme Accounting Service runs the updates at 9 pm every day and updates member status.
  2. The membership status is reevaluated and crosschecked.
  3. Some members are suspended, and some are reinstated.
  4. The updates are pushed to the database.
Extensions:
  1. Error is raised if a member does not exist.
  2. The suspended members are informed of their overdue fees, and reinstated members are informed of their status.
Use Case: Edit provider records
Context: During the day, the computer is set into interactive mode for the operator to edit provider records.
Actors: Operator
Main Success Scenario:
  1. Operator intends to change the provider records.
  2. The operator is displayed current provider info and prompted for change of records.
  3. The changes are verified and reconfirmed by the operator.
  4. The changes are saved to the database.
  5. The operator is informed of changes and taken to interactive operations for further commands.
Use Case: Delete provider
Context: During the day, the computer is set into interactive mode for the operator to delete provider
Actors: Operator
Main Success Scenario:
  1. Operator intends to delete provider via the Interactive Operations.
  2. Operator is displayed info about the provider and prompted to delete said provider.
  3. The deletion of the provider must be verified and reconfirmed by the operator.
  4. Provider is deleted from database and operator is informed of it.
  5. Operator is taken back to interactive operations.
Use Case: Add provider
Context: During the day, the computer is set into interactive mode for the operator to add providers
Actors: Operator
Main Success Scenario:
  1. Operator intends to add provider via the Interactive Operations
  2. Operator is asked to add information about the provider
  3. The addition of the provider is verified by the operator
  4. The provider is added to the database, and the operator is informed of it
  5. Operator is taken back to interactive operations
